# מדריך משתמש ומסמך טכני - CompareD

מסמך זה משמש כמדריך טכני ומדריך משתמש מפורט עבור מערכת השוואת הנתונים הדינמית CompareD. המדריך מפרט את זרימת העבודה של המשתמש בממשק מול הפעולות הטכניות המתבצעות מאחורי הקלעים בבקר השרת.

---

## 1. מסך הגדרת חיבורי מסדי נתונים (דף הבית)

דף הבית הוא נקודת הכניסה למערכת שבה המשתמש מגדיר את פרטי החיבור של שתי סביבות העבודה או בוחר לעבור למסלול השוואת קבצים עצמאיים.

### מדריך למשתמש:
1. **מה רואים במסך**: שתי תיבות בחירה בתוך פריסת Bento Grid מעוצבת. התיבה הימנית מיועדת עבור SQL Server (מקור) והתיבה השמאלית עבור Oracle Database (יעד). הפרופילים נטענים אוטומטית מתוך קובץ ההגדרות של השרת.
2. **מה בוחרים**: בוחרים פרופיל חיבור לכל אחד מהצדדים. המערכת כוללת פרופיל מובנה בשם `MockConnectionString` המאפשר הרצת הדגמה (Mock Data) של מסכי המערכת ללא תלות במסדי נתונים פיזיים.
3. **היכן לוחצים**: לוחצים על הכפתור הזוהר "התחבר והבא טבלאות".
4. **מסלול חלופי**: בתחתית העמוד מופיע כרטיס רחב בשם "השוואה ישירה של קובצי CSV / Excel". לחיצה על כפתור "📂 השוואת קובצי CSV" תעביר את המשתמש למסך העלאת קבצים ישירה ללא תלות במסדי נתונים.
5. **ניהול שגיאות**: במידה והחיבור נכשל, הדף ייטען מחדש ויציג תיבת שגיאה אדומה המפרטת את סיבת השגיאה, תוך שמירה על פרופילי החיבור שהוזנו.

### מדריך טכני למפתח:
1. **בקר ופעולה**: הבקשה נשלחת כשיטת POST אל הפעולה `Connect` בבקר `CompareController.cs`.
2. **פעילות מאחורי הקלעים**: הבקר מזהה את מחרוזות החיבור המתאימות לפרופילים שנבחרו. במידה והמחרוזת היא `MockConnectionString`, הבקר מדמה חיבור ומחזיר רשימת טבלאות דמי (`USERS`, `ORDERS`, `PRODUCTS`). בחיבורים אמיתיים, הבקר פותח חיבורים במקביל (Task.WhenAll) באמצעות `SqlConnection` ו-`OracleConnection` ושולף את שמות הטבלאות והתצוגות (Views) הזמינות.
3. **ניהול סשן ומעבר שלב**: אם החיבור הצליח, מחרוזות החיבור נשמרות בסשן השרת (`HttpContext.Session`). הבקר יוצר מודל מסוג `TableSelectionViewModel` ומפנה לתצוגה `SelectTables.cshtml`.

## 2. מסך בחירת טבלאות ותצוגות להשוואה

מסך זה מאפשר למשתמש לבחור אילו טבלאות או תצוגות (Views) הוא מעוניין להשוות בין שתי המערכות.

### מדריך למשתמש:
1. **מה רואים במסך**: שתי תיבות בחירה (Dropdowns) המציגות את רשימת הטבלאות והתצוגות שנשלפו. מעל כל תיבת בחירה מופיעה תיבת טקסט לחיפוש מהיר.
2. **מה עושים**:
   - מקלידים בתיבת החיפוש המהיר כדי לסנן באופן מיידי את רשימת האובייקטים לפי שמם (הסינון מתבצע בזמן אמת ומסמן אוטומטית את התוצאה הראשונה).
   - בוחרים טבלה או תצוגה מתיבת הבחירה לכל אחד מהצדדים.
3. **ייבוא תצורת מיפוי מוכנה (JSON)**: אם המשתמש כבר ביצע השוואה בעבר ושמר את קובץ הגדרות המיפוי (JSON), ניתן לטעון אותו בתיבה הימנית וללחוץ על "📂 טען והרץ השוואה" כדי להריץ את הבדיקה ישירות ללא מעבר בשלבים הבאים.
4. **השוואת קובצי CSV / Excel**: בתיבה השמאלית של שלב הייבוא, ניתן ללחוץ על "📂 ייבוא קבצים והשוואה" למעבר מהיר למסך העלאת קבצים (CSV / Excel).
5. **היכן לוחצים**: לוחצים על הכפתור הזוהר "המשך לסקירת סכמה" בתחתית הדף.

### מדריך טכני למפתח:
1. **בקר ופעולה**: שליחת הטופס מפעילה את הפעולה `SchemaReview` בבקר `CompareController.cs` כשיטת POST.
2. **שחזור ומיפוי JSON**: כפתור ייבוא ה-JSON מפעיל פונקציית JavaScript בשם `importMappingConfig()`. הפונקציה מפרשת את קובץ ה-JSON, יוצרת טופס מוסתר בצורה דינמית ושולחת אותו לפעולה `CompareData` עם כל המיפויים מוכנים מראש.

## 3. מסך סקירת סכמה (של מסדי נתונים)

מסך זה מציג השוואה מבנית מלאה בין שתי הטבלאות שנבחרו (SQL Server מול Oracle) לפני הרצת השוואת הנתונים.

### מדריך למשתמש:
1. **מה רואים במסך**:
   - השוואה מבנית מפורטת בטבלה: שמות העמודות, סוגי הנתונים (ב-SQL Server וב-Oracle) וחיווי האם העמודה קיימת בשני הצדדים או חסרה באחד מהם.
   - עמודת מפתח ראשי מוצעת: המערכת מזהה ומציעה באופן אוטומטי את עמודת המפתח הראשי המשותפת.
   - שמירת מיפוי סכמה: כפתור ייחודי המאפשר לשמור את המבנה והמיפוי הנוכחי לקובץ JSON חיצוני לצורך שימוש חוזר עתידי.
2. **היכן לוחצים**: לוחצים על הכפתור הזוהר "המשך למיפוי שדות והשוואה".

### מדריך טכני למפתח:
1. **בקר ופעולה**: הפעולה `CompareSchemaAsync` בשירות `CompareService.cs` שולפת את נתוני הסכמות והמפתחות הראשיים משני מסדי הנתונים במקביל באמצעות שאילתות קטלוגיות מאובטחות (מתוך `INFORMATION_SCHEMA.COLUMNS` ומתוך `USER_TAB_COLUMNS`).
2. **שמירת קובץ JSON**: פונקציית JavaScript בצד הלקוח יוצרת אובייקט Blob ומורידה קובץ תצורה במבנה הבא:
   ```json
   {
     "sqlTable": "USERS",
     "oracleTable": "USERS",
     "sourceFields": ["ID", "NAME", "EMAIL"],
     "targetFields": ["ID", "NAME", "EMAIL"],
     "fieldRoles": ["Key", "Compare", "Compare"]
   }
   ```

## 4. ממשק השוואת קובצי נתונים (CSV / Excel XLSX)

מסלול זה מאפשר העלאה של שני קובצי CSV או Excel XLSX עצמאיים, סקירת המבנה שלהם, בחירת מפתח ראשי, התאמת עמודות להשוואה, והרצת ההשוואה בזיכרון השרת.

### א. שלב 1: העלאת הקבצים
- **מדריך למשתמש**: המשתמש מעלה שני קבצים (Table 1 כמקור ו-Table 2 כיעד). המערכת תומכת בקובצי `.csv` וכן בקובצי `.xlsx` של אקסל. לאחר בחירת הקבצים, שמם יופיע בירוק בתוך קוביית ההעלאה. לחיצה על "הרץ בדיקת השוואה" תעביר אותו למסך המיפוי והסקירה.
- **מדריך טכני למפתח**: הבקשה נשלחת כ-POST לפעולה `CompareUploadedFiles` המאמתת שהקבצים אינם ריקים ושהסיומת שלהם תקינה. הקבצים נשמרים תחת נתיבים ייחודיים בתיקיית `temp_uploads` בשרת, ונתיביהם נשמרים ב-Session. המשתמש מופנה ל-GET `CompareFilesSchemaReview`.

### ב. שלב 2: סקירת סכמה ומיפוי עמודות (CSV/Excel Review)
- **מדריך למשתמש**: 
  - המשתמש רואה את רשימת עמודות המקור שזוהו בקובץ הראשון.
  - המשתמש בוחר מתוך תיבת בחירה את עמודת המפתח הראשי של המקור (העמודה שתשמש לזיהוי והתאמה חד ערכית בין השורות).
  - המשתמש רואה טבלה ובה תיבות סימון (V) - הוא יכול לסמן או להסיר סימון מעמודות שאינו מעוניין להשוות כלל.
  - עבור כל עמודת מקור מסומנת, המשתמש יכול לבחור מה dropdown שלצידה איזו עמודת יעד בקובץ השני מתאימה לה (המערכת ממפה מראש שמות זהים בצורה אוטומטית).
  - בתחתית, המשתמש מגדיר את כמות הרשומות המקסימלית לעיבוד (Max Rows) ולוחץ על הכפתור "הרץ השוואת נתונים".
- **מדריך טכני למפתח**:
  - GET `CompareFilesSchemaReview` שולף את נתיבי הקבצים מהסשן, ומפעיל את `CsvParser.GetHeaders(path)` כדי לחלץ את שמות העמודות.
  - עבור קובצי `.csv` נעשה שימוש במפרש טקסט.
  - עבור קובצי `.xlsx` נעשה שימוש ב-`MiniExcel.Query` כדי לשלוף את שמות העמודות מתוך גיליון האקסל בצורה יעילה במיוחד ללא קריאת כל הקובץ לזיכרון.
  - המידע נשלח לתצוגה `CompareFilesSchemaReview.cshtml` המרנדרת את בקרת המיפוי.
  - שליחת הטופס פונה ל-POST `RunFilesComparison` המקבל את הגדרות ה-PrimaryKey, רשימת ה-Source fields שנבחרו, והמיפויים מתוך `Request.Form`.
  - הנתונים נשלפים מהקבצים בעזרת `CsvParser.ParseFile(path)`, מוגבלים לפי `maxRows`, ומועברים ל-`CompareInMemoryDatasets` המבצע את ההשוואה ומציג את הדשבורד `Results`.

## 5. מסך תוצאות ההשוואה וייצוא דוחות

דשבורד המציג את הממצאים האנליטיים בצורה מפורטת, לצד אפשרויות הורדה וייצוא.

### מדריך למשתמש:
1. **מה רואים במסך**:
   - **ארבעה כרטיסי סיכום (Bento Grid)**: רשומות זהות, הפרשי נתונים, חוסרים ביעד וחוסרים במקור.
   - **כרטיסיות (Tabs)**: מעבר נוח בין פירוט הבדלי הנתונים (המוצגים בטבלה שבה התאים השונים מודגשים באדום עדין), רשימת מפתחות חסרים, וכפילויות מפתח שהוחרגו.
2. **ייצוא לאקסל (Excel)**: לחיצה על כפתור "ייצוא לאקסל (CSV)" תוריד קובץ CSV מסודר הכולל את כל סיכומי הממצאים ופירוט ההבדלים. הקובץ מקודד עם תווית UTF-8 BOM מיוחדת התומכת בהצגה תקינה של תווים בעברית בתוך תוכנת Excel.
3. **ייצוא לוורד (Word)**: לחיצה על כפתור "ייצוא לוורד (Doc)" תוריד קובץ מסמך מעוצב עם טבלאות וסטייל מובנה המפרט את הדוח הניהולי המלא.
4. **היכן לוחצים**: לוחצים על "התחל השוואה חדשה" כדי לחזור לדף הבית.

### מדריך טכני למפתח:
1. **ייצוא Excel (CSV)**: מבוצע בצד הלקוח באמצעות יצירת מחרוזת מופרדת בפסיקים הכוללת את סיכום הרשומות ופרטי התאים השונים. המחרוזת מומרת למערך בתים עם ה-BOM של UTF-8 (`\uFEFF`) ונשלחת להורדה כסוג תוכן `text/csv;charset=utf-8;`.
2. **ייצוא Word (HTML-Doc)**: מבוצע בצד הלקוח על ידי בניית מסמך HTML תקני הכולל הגדרות CSS ייעודיות לטבלאות מעוצבות, כותרות, וצבעי רקע. מסמך ה-HTML נעטף ב-Blob עם סוג תוכן `application/msword` ונשמר כקובץ `.doc`.
3. **קיבוץ בעיות לפי תבניות (Discrepancy Patterns)**: אלגוריתם השרת מקבץ שורות שונות לפי הרכב העמודות השונות שלהן ומציג דוגמאות לכל תבנית, מה שמאפשר למשתמש לזהות בעיות רוחביות במהירות.

## 6. דגשי אבטחה, ביצועים ועבודה בלתי פולשנית (Zero-Impact)

במסגרת תכנון הארכיטקטורה של CompareD, יושמו דגשי אבטחה וביצועים קפדניים בתהליך שליפת הנתונים והשוואתם כדי להבטיח עבודה בטוחה ובלתי פולשנית:

1. **מנגנון Read-Only מוחלט**: הקוד מוגדר ברמת הארכיטקטורה לביצוע פעולות `SELECT` בלבד. אין במערכת שום הרשאה או לוגיקה המאפשרת ביצוע פעולות `INSERT`, `UPDATE` או `DELETE` בבסיסי הנתונים שלכם.
2. **עבודה בזיכרון השרת (In-Memory Processing)**: המערכת אינה יוצרת טבלאות זמניות בתוך מסד הנתונים ואינה מבצעת עליהם פעולות כתיבה או מחיקה. שליפת הנתונים מתבצעת ישירות לתוך משתני זיכרון (`Dictionaries`) בשרת האפליקציה. כל תהליך ההשוואה, הנרמול וניתוח ההבדלים מתבצע ב-RAM של השרת בלבד.
3. **מניעת הזרקות (SQL Injection Prevention)**: השאילתות למסדי הנתונים מבוצעות באמצעות Parameterized Queries בלבד. נתוני המשתמש (כמו שמות טבלאות או תנאי סינון) עוברים תהליך של טיהור (Sanitization) קפדני לפני שילובם בשאילתה, מה שמונע לחלוטין כל ניסיון להזרקת קוד חיצוני.
4. **מגבלות עומס וניהול משאבים (Resource Management)**: כדי למנוע השפעה על ביצועי בסיסי הנתונים הארגוניים בזמן השליפה, הוטמעה הגבלת רשומות דינמית (`Max Rows`) המאפשרת שליטה מלאה על כמות הנתונים הנטענת בכל פעולה, בהתאם ליכולות השרת ובסיסי הנתונים.

גישה זו מבטיחה עבודה במודל **Zero-Impact** על הנתונים הארגוניים תוך שקיפות מלאה מול מבנה הסכמות שלכם.
